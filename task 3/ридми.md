# AdScale: стратегия данных, масштабирование и отказоустойчивость

## Цель документа

Спроектировать стратегию работы с данными в целевой микросервисной архитектуре AdScale, обеспечив горизонтальное масштабирование, отказоустойчивость и соблюдение строгих требований по латенции (P95 ≤ 100 мс) при росте нагрузки до 50 000 RPS.

## Ключевые решения

- **Database per Service** – каждый микросервис владеет своей базой данных (или схемой), что устраняет конкуренцию за ресурсы и позволяет независимо масштабировать хранилища.
- **Разделение OLTP и аналитики** – оперативная база (PostgreSQL) и аналитическое хранилище (ClickHouse) работают независимо.
- **Кэширование через Redis** – для кампаний, ставок, бюджетов. Горячие данные вынесены из PostgreSQL.
- **Потоковая обработка через Kafka** – все события (показы, клики) проходят через единую шину событий.
- **Отказоустойчивость** – репликация, резервные копии, RPO/RTO определены для каждого сервиса.


## Принятые ADR

| ID | Решение | Обоснование |
|----|---------|-------------|
| ADR-010 | Bidding Service — Redis как основное хранилище | Требуется сверхбыстрый доступ (<5 мс), данные не требуют durability (можно восстановить из Campaign Service) |
| ADR-011 | Campaign Service — PostgreSQL | ACID-транзакции, сложные запросы, долговременное хранение |
| ADR-012 | Stats Service — ClickHouse | Огромный поток событий (миллионы в секунду), нужна аналитика по времени, колоночное хранение |
| ADR-013 | Financial Service — PostgreSQL с репликацией | Требуется строгая согласованность, WAL-репликация для отказоустойчивости |
| ADR-014 | Analytics Service — read replica PostgreSQL + ClickHouse | Отдельные отчёты не влияют на OLTP |
| ADR-015 | CQRS для Campaign Service | Команды (CRUD) идут в PostgreSQL, запросы (чтение для Bidding) через Redis (read model) |
