# Стратегии масштабирования и отказоустойчивости баз данных

## Репликация Master-Slave

**Где применяем**:

- **PostgreSQL для Campaign Service** – один master, два асинхронных slave (в разных availability зонах). Чтения (например, для Analytics) идут на slave.
- **PostgreSQL для Financial Service** – master + синхронная реплика (для нулевой потери данных) и асинхронная для отчётности.
- **Redis Cluster** – встроенная репликация: каждый мастер имеет одну или несколько реплик.

**Зачем**: высокая доступность (failover при падении master), распределение нагрузки на чтение.

## Шардирование (горизонтальное партиционирование)

### Campaign Service

**Ключ шардирования**: `campaign_id` (хэш по модулю N).

**Почему**: большинство запросов идут по конкретному `campaign_id` (чтение кампании по ID). Это равномерно распределяет нагрузку.

**Реализация**: PostgreSQL 11+ с декларативным партиционированием по хэшу (или Citus). 8 шардов (можно увеличивать позже).

### Stats Service (ClickHouse)

**Ключ шардирования**: `date` + `event_type` (показ/клик).

**Почему**: запросы почти всегда содержат фильтр по дате. Данные старше 30 дней можно перемещать в холодное хранилище (S3) через TTL.

**Реализация**: распределённая таблица ClickHouse с шардированием по дате (интервалами по дню). 12 шардов (на каждый месяц года – избыточно, но позволяет перебалансировать).

### Bidding Service (Redis Cluster)

**Ключ шардирования**: `campaign_id` (хэш).

**Реализация**: встроенное шардирование Redis Cluster (16384 слота). Достаточно 4 мастер-нод (по 4096 слотов каждая) и 4 реплик.

## Read replicas для аналитики

**Где**: 
- Одна read replica PostgreSQL для Campaign Service – используется Analytics Service для получения названий кампаний без блокировки master.
- Для Financial Service – replica не требуется, так как отчётов по финансам мало, можно читать с master (низкая нагрузка).

**Почему**: отделяем тяжёлые аналитические запросы от оперативной записи.

## CQRS (разделение команд и запросов)

**Применяем для Campaign Service**:

- **Command side (write)**: PostgreSQL master. Принимает CRUD-операции от рекламодателя.
- **Query side (read)**: Redis (через Bidding Service). Campaign Service при каждом изменении публикует событие `CampaignUpdated` в Kafka. Bidding Service слушает событие и обновляет свой Redis-кэш.

**Преимущества**:
- Bidding Service не читает из PostgreSQL в hot path.
- Можно масштабировать read-модель независимо (добавлять больше Redis реплик).
- Согласованность – eventual consistency (задержка менее секунды), что приемлемо для ставок (допустима устаревшая кампания на несколько секунд).

**Для Financial Service CQRS не применяем** – нужна строгая согласованность.

## Общая схема масштабирования

- Bidding Service и Auction Engine – stateless, масштабируются горизонтально (K8s HPA).
- Redis Cluster – масштабируется добавлением шардов (перебалансировка слотов).
- ClickHouse – масштабируется добавлением шардов, данные перераспределяются автоматически.
- PostgreSQL (Campaign) – вертикально до 8 vCPU, затем шардирование через Citus.
