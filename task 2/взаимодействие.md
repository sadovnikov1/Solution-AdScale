# Схема взаимодействия RTB-компонентов

## Общая архитектура взаимодействия

DSP-партнёр → API Gateway → Bidding Service → Auction Engine → Delivery Service

События (клики, показы) идут асинхронно: Delivery Service → Stats Writer → Kafka → ClickHouse / Analytics

## Выбор протоколов: обоснование

### Внешний протокол: OpenRTB / HTTPS

DSP-партнёр работает по стандарту OpenRTB (протокол RTB-аукционов). AdScale не может его изменить. Запросы идут по HTTPS, тело — JSON. Формат специфицирован IAB Tech Lab. Требования: время ответа ≤ 100 мс.

**Принятое решение:** Использовать OpenRTB 2.5 как есть. API Gateway будет проксировать запросы, не меняя формат.

### Внутренний протокол: gRPC (Bidding ↔ Auction)

**Почему gRPC, а не REST.**

Первое — производительность. gRPC использует Protobuf (бинарный формат) вместо JSON. Сериализация/десериализация быстрее, размер сообщения меньше. Это критично для 18 000 RPS.

Второе — строгие контракты. .proto-файлы задают жёсткую схему, исключая ошибки с типами полей.

Третье — HTTP/2. Мультиплексирование запросов, сжатие заголовков, более эффективное использование соединений.

Четвёртое — встроенные таймауты и cancellation. gRPC поддерживает контекстные таймауты на уровне протокола.

**Почему не REST.** JSON сериализация медленнее, нет строгих контрактов (требуется отдельная валидация), HTTP/1.1 без мультиплексирования требует больше соединений.

**Принятое решение:** Bidding Service и Auction Engine общаются через gRPC. Метод — `EvaluateCandidates`.

### Протокол для событий: Kafka

**Почему Kafka, а не синхронный HTTP.**

Первое — сглаживание пиков. Клики и показы приходят неравномерно (ночные низы, дневные пики). Kafka буферизует события, не теряя их.

Второе — развязка сервисов. Stats Writer пишет в Kafka и падает. ClickHouse consumer может восстанавливаться независимо.

Третье — гарантия доставки. Kafka хранит сообщения на диске и реплицирует. При падении consumer’а события не теряются.

Четвёртое — replay. При ошибках в аналитике можно перечитать топик с начала.

**Почему не синхронный HTTP.** При пике 18 000 кликов в секунду синхронная запись создаст такую же нагрузку на БД, как и сейчас. Буферизация невозможна.

**Принятое решение:** Все события (показы, клики, конверсии) публикуются в Kafka. ClickHouse потребляет топик через Kafka Connect или самописный consumer.

### Протокол для управления кампаниями: gRPC + Kafka

Campaign Service общается с Bidding Service асинхронно через Kafka:

1. Рекламодатель меняет кампанию через API Gateway → Campaign Service пишет в PostgreSQL
2. Campaign Service публикует событие `CampaignUpdated` в Kafka
3. Bidding Service потребляет событие и обновляет Redis-кэш

В hot path (обработка bid request) Bidding Service никогда не вызывает Campaign Service синхронно.

## Нужен ли API Gateway?

Да, нужен. Без него пришлось бы решать несколько проблем самостоятельно:

- **Аутентификация** DSP-партнёра (API-ключи, JWT, mTLS) — единое место
- **Rate limiting** — защита от превышения лимитов, штрафов
- **Маршрутизация** — разные DSP могут иметь разные эндпоинты
- **Circuit Breaker** — единая защита от падения Bidding Service
- **Мониторинг** латенции на входе

**Выбор технологии:** Kong (open source, высокая производительность, поддержка OpenResty/Nginx). Альтернативы — Traefik (легче) или NGINX (ручная конфигурация).

API Gateway ставится перед Bidding Service. Он ничего не знает о бизнес-логике, только маршрутизирует и ограничивает.

## Диаграмма последовательности (sequence.puml)

Ниже код диаграммы для PlantUML. Скопируйте в файл `diagrams/bidding-sequence.puml`.

```plantuml
@startuml
title Последовательность обработки Bid Request (RTB)

actor "DSP Partner" as DSP
participant "API Gateway\n(Kong)" as Gateway
participant "Bidding Service" as Bidding
participant "Redis Cluster" as Redis
participant "Auction Engine" as Auction
participant "Kafka" as Kafka
database "ClickHouse" as CH

DSP -> Gateway: POST /rtb/bid\n(OpenRTB)
activate Gateway

Gateway -> Gateway: Rate limiting\nAuth (API Key)
Gateway -> Bidding: Forward bid request (HTTP/gRPC)
activate Bidding

Bidding -> Redis: GET campaign:{id}\n(для каждого кандидата)
Redis --> Bidding: Campaign data
Bidding -> Bidding: Фильтрация по таргетингу\nВычисление ставки

alt Есть подходящие кампании
    Bidding -> Auction: EvaluateCandidates (gRPC)
    activate Auction
    Auction --> Bidding: Winner & price
    deactivate Auction
else Нет кампаний
    Bidding -> Bidding: Формирует no-bid
end

Bidding --> Gateway: Bid response (JSON)
deactivate Bidding
Gateway --> DSP: OpenRTB response
deactivate Gateway

Bidding -> Kafka: Publish BidRequestReceived (async)
Kafka --> CH: Consume (streaming)

@enduml
