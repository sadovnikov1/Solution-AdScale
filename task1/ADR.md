3. Архитектурные решения (ADR) для целевого состояния
   
ADR-001: Выделение ClickHouse для событийного хранилища

Контекст: Статистика кликов/показов пишется синхронно в PostgreSQL, создавая write-конфликты. При росте до 50 000 RPS запись в PostgreSQL станет невозможной. Решение: Использовать Kafka для буферизации и ClickHouse как колоночное хранилище для событий. Аргументы: ClickHouse оптимизирован для высокоскоростной вставки (миллионы событий/сек), поддерживает дедупликацию. Kafka сглаживает пики. Последствия: Появление eventual consistency для отчётов (задержка 1–2 секунды) – приемлемо. Требуется обучение команды ClickHouse.

ADR-002: Вынос аналитики на read replica + Data Warehouse

Контекст: Analytics Service выполняет тяжёлые SQL-запросы на production-БД, вызывая блокировки. Решение: Создать read replica PostgreSQL для отчётов, а для исторической аналитики использовать ClickHouse. Аргументы: Read replica не влияет на OLTP-производительность. ClickHouse позволяет строить быстрые агрегаты. Последствия: Увеличение стоимости хранения (дополнительная реплика). Необходим ETL-процесс для переноса данных из OLTP в ClickHouse (можно через Kafka Connect).

ADR-003: Кэширование ставок и кампаний в Redis

Контекст: Auction Engine синхронно читает БД для каждого bid request, что добавляет латенцию 20–30 мс. Решение: Загружать все активные кампании и ставки в Redis при старте и обновлять через CDC (Change Data Capture) или инвалидацию по TTL. Аргументы: Redis обеспечивает доступ за <1 мс, горизонтально масштабируется (кластер). Последствия: Дополнительная сложность синхронизации кэша. При сбое Redis – fallback на БД с degraded mode.

ADR-004: Контейнеризация и оркестрация (Kubernetes)

Контекст: Ручной деплой, отсутствие горизонтального масштабирования. Решение: Упаковать каждый микросервис в Docker, оркестрировать через Kubernetes (managed, например, EKS или GKE). Аргументы: Автоматическое масштабирование (HPA), самовосстановление, rolling updates. Последствия: Требуется DevOps-навыки (в штате есть DevOps-инженеры). Увеличение потребления ресурсов на инфраструктуру.

ADR-005: Разделение базы данных по доменам (Database per service)

Контекст: Одна PostgreSQL инстанция для всех сервисов – главный bottleneck. Решение: Выделить отдельные БД (или схемы с изоляцией) для Campaign Service, Payment Service. Для статистики использовать ClickHouse. Для аналитики – отдельную read replica. Аргументы: Устранение конкуренции за ресурсы. Каждый сервис может масштабировать своё хранилище независимо. Последствия: Сложность поддержания согласованности между сервисами (использовать Saga или eventual consistency). Дополнительные затраты на администрирование
