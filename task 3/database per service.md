# Стратегия Database per Service

## Общие принципы

- Каждый микросервис владеет **своим** хранилищем данных. Никакой сервис не обращается к базе другого сервиса напрямую.
- Доступ к данным возможен только через API сервиса-владельца.
- Для критичных по латенции сервисов (Bidding) используется кэш Redis как основное хранилище (с возможностью восстановления из source of truth).
- Аналитические и событийные данные вынесены в ClickHouse.

## Выбор базы данных по сервисам

### Сервис ставок (Bidding Service)

**Тип БД:** Redis Cluster (in-memory key-value)

**Обоснование**:
- Требование по латенции <5 мс на чтение данных кампании.
- Данные имеют высокую частоту чтения (каждый bid request) и низкую частоту записи (обновление кампаний – раз в минуты/часы).
- Redis обеспечивает репликацию (master-slave) и шардирование по ключу.
- Данные не критичны с точки зрения durability – при полной потере Redis можно восстановить из Campaign Service (source of truth) за несколько минут.

**Хранимые сущности**: активные кампании, ставки, таргетинги, остатки бюджетов (кэшированные копии).

### Сервис кампаний (Campaign Service)

**Тип БД:** PostgreSQL

**Обоснование**:
- Требуются ACID-транзакции при создании/изменении кампаний и финансовых лимитов.
- Сложные запросы для отчётов (фильтрация по множеству полей).
- Данные являются source of truth для всех остальных сервисов.
- Поддерживает репликацию master-slave и шардирование (через расширение Citus или ручное партиционирование).

**Хранимые сущности**: кампании, креативы, правила таргетинга, бюджеты, ставки (полная история изменений).

### Сервис статистики (Stats Service)

**Тип БД:** ClickHouse

**Обоснование**:
- Огромный поток событий: до 50 000 показов/кликов в секунду.
- Колоночное хранение позволяет эффективно сжимать данные и выполнять агрегации за миллисекунды.
- Вставка данных идёт пачками (через Kafka) – ClickHouse оптимизирован для высокоскоростной записи.
- Не требуется обновление уже записанных событий (только чтение и агрегация).
- Поддерживает распределённые таблицы для горизонтального масштабирования.

**Хранимые сущности**: события показов, кликов, конверсий (денормализованные, с временной меткой).

### Сервис аналитики (Analytics Service)

**Тип БД:** PostgreSQL (read replica) + ClickHouse

**Обоснование**:
- Отчёты рекламодателей требуют как агрегированных данных из ClickHouse (быстро), так и текущих данных кампаний из PostgreSQL (названия, бюджеты).
- Использование read replica PostgreSQL исключает блокировки на master.
- ClickHouse используется для «сырых» событий и глубокого анализа.
- Можно применять материализованные представления в ClickHouse для ускорения типовых отчётов.

**Хранимые сущности**: агрегированные отчёты (кэшируются), витрины данных.

### Сервис финансов (Financial Service)

**Тип БД:** PostgreSQL

**Обоснование**:
- Строгая согласованность и ACID – критично для списания средств.
- Необходима поддержка двухфазного коммита (или Saga) при взаимодействии с платёжным шлюзом.
- WAL (Write-Ahead Log) используется для репликации и аудита.
- Данные долговременного хранения.

**Хранимые сущности**: счета рекламодателей, транзакции, инвойсы.

### Итоговая таблица выбора (текстово)

**Bidding Service** – Redis Cluster (in-memory, key-value) – производительность, низкая латенция, допустима потеря.

**Campaign Service** – PostgreSQL – ACID, сложные запросы, source of truth.

**Stats Service** – ClickHouse – высокая скорость записи, колоночное хранение, аналитика.

**Analytics Service** – PostgreSQL read replica + ClickHouse – изоляция от OLTP, быстрые отчёты.

**Financial Service** – PostgreSQL – строгая согласованность, ACID, аудит.
