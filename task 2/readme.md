# AdScale RTB-интеграция с DSP — проектирование

## Цель задания

Спроектировать архитектуру критичного компонента — RTB-сервиса для интеграции с новой DSP-площадкой, обеспечив соблюдение жёстких требований по времени отклика (P95 ≤ 100 мс) и надёжности при росте нагрузки до 18 000 RPS.

## Основные результаты

1. **Выделение Bidding Service из монолита** — описание границ, API, зависимостей и модели данных.
2. **Выбор протоколов взаимодействия** — внешний OpenRTB (HTTPS), внутренний gRPC для синхронных вызовов, Kafka для асинхронных событий.
3. **API Gateway для DSP** — маршрутизация, rate limiting, аутентификация, Circuit Breaker, мониторинг.
4. **Паттерны надёжности** — Circuit Breaker, retry с экспоненциальной задержкой, идемпотентность, fallback-стратегии.

## Структура документов

- `bidding-service.md` — спецификация сервиса ставок
- `interaction.md` — схема взаимодействия и обоснование протоколов
- `api-gateway.md` — дизайн API Gateway для DSP
- `diagrams/bidding-sequence.puml` — диаграмма последовательности обработки bid request
- `diagrams/component.puml` — компонентная диаграмма RTB-сервиса

## Ключевые архитектурные решения (ADR)

| ID | Решение | Обоснование |
|----|---------|-------------|
| ADR-006 | Bidding Service — отдельный сервис (Go) | Высокая производительность, низкая латенция, независимое масштабирование |
| ADR-007 | gRPC для внутренних вызовов (Bidding ↔ Auction) | Низкая латенция, бинарный протокол, строгие контракты |
| ADR-008 | Kafka для событий (клики/показы) | Асинхронность, сглаживание пиков, гарантия доставки |
| ADR-009 | Circuit Breaker для вызовов Campaign Service | Защита от каскадных отказов, fallback через Redis кэш |
